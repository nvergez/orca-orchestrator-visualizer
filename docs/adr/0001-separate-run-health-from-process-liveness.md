# Separate run health from convergence and Orca process liveness

`Run.live` collapsed three different claims: whether task work had converged, whether the run had recent activity evidence, and whether the Orca process was running. We will model convergence and `lastActivityAt` as server-derived run facts, derive `active | silent | finished` run health from those facts and the client's wall clock, and keep `Meta.liveness` as the independent process signal. This avoids presenting an abandoned dispatch as active merely because Orca is open, lets a blocked or pending run remain unfinished, and lets health cross the ten-minute recency boundary without requiring a database write or SSE event.

The wire change is additive first: `Run` gains `converged` and `lastActivityAt`; deprecated `endedAt` remains an exact alias of `lastActivityAt`, and deprecated `live` remains a snapshot-time compatibility projection equivalent to `meta.liveness === 'live' && runHealth(run, snapshotNow) === 'active'`. New clients must ignore both deprecated fields. They may be removed only with a versioned breaking wire contract, not during the #48 implementation.

Last activity is the maximum readable timestamp across every task's creation and completion and every dispatch attempt's dispatch, completion, last heartbeat, and last failure. Messages do not extend the bound: task-id-less message attribution already depends on that bound, so using attributed messages as input would make attribution recursive and could let a chain of weak handle matches extend itself indefinitely. The handle-attribution window remains six hours after activity, but its tail moves from `endedAt + 6h` to `lastActivityAt + 6h`; direct valid `payload.taskId` attribution and the existing ambiguous-to-null rule are unchanged.
