# Notify from the attention queue, against a per-connection baseline

Reaching a supervisor who is not looking at the page must not create a second urgency model. The document title, the favicon and any desktop notification are therefore derived from the attention queue (#56) and from nothing else: the tab shows the queue's count and a badged favicon, and a notification is one attention item delivered. Nothing else in the client classifies raw gates, dispatches or failures as urgent, so the tab, the queue and the rail cannot disagree about what needs intervention or about how much of it there is.

The queue present on the first snapshot of a stream connection — on initial load, and again after every `EventSource` reconnect — is the notification baseline: its items are recorded as seen and never announced. The database is never pruned, so an unpruned four-day file full of blocking gates would otherwise burst onto the desktop at every page load, and a dropped connection replays everything that happened while the page was blind (SPEC §6.2). The accepted cost is explicit: **a cause that first appears while the page is disconnected is never notified**, because on reconnect it is indistinguishable from history. It still enters the queue, still counts on the tab, and is still one click from its task. A notifier that announced it would be a notifier that announces a laptop lid.

A notification is sent at most once per attention item identity, and the identity is #56's stable id — built from the durable rows behind the evidence, never from array positions or the clock. Repeated snapshots, re-rankings, tab focus changes and a cause whose evidence leaves the queue and returns are all the same id, and none of them announces twice.

Delivery is opt-in and stored locally, default off, and the browser's permission is requested only from an explicit user gesture. Permission that is denied, revoked mid-session, absent (no `Notification` API), or present but unusable (Chrome on Android throws from the constructor, wanting a service worker that #60 puts out of scope) degrades to the attention tab state with no error: the opt-in and the permission are two separate facts, and the control names which of them is the reason it cannot deliver, because only one of them is something the reader can go and change.
